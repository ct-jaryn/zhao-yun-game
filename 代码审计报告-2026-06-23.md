## 1. 严重问题 🔴

### 1.1 `__WHITE` 纹理不存在，fallback 分支必然异常

**位置**：`entities/Player.js:182`、`entities/Enemy.js:104`、`entities/DiaoChan.js:31`

```js
this.sprite = this.scene.add.sprite(this.x, this.y, '__WHITE');
```

**问题**：Phaser 3.80 默认内置纹理只有 `__DEFAULT`、`__MISSING`，**没有 `__WHITE`**。这三处 fallback 在「目标切片纹理不存在」时触发，会创建一个引用不存在纹理的 sprite。Phaser 对缺失纹理会渲染为绿块（`__MISSING`）并持续打 console 警告，`setDisplaySize`/`setTint` 在 `__MISSING` 上行为不稳定。

**触发条件**：任意玩家/敌人/貂蝉切片资源加载失败（网络波动、路径错误、`PreloadScene` 的 `loaderror`）。历史报告声称资源「✅ 完整」，但 `PreloadScene.js:14-17` 仍保留 `loaderror` 收集逻辑，说明加载失败是预期场景，fallback 必须可用。

**修复建议**：在 `BootScene.create()` 中预生成白色纹理：
```js
this.textures.generate('__WHITE', { data: ['0xffffff'], width: 1, height: 1 });
```
或统一改用 Phaser 内置 `__DEFAULT`。

### 1.2 Boss 首次死亡后 60 秒不淡出，满透明度「躺尸」

**位置**：`entities/Enemy.js:162-166`（设 `deathTimer=60`）、`entities/Enemy.js:342`（`progress = deathTimer / 0.6`）

**问题**：Boss 首次死亡进入复活流程时：
```js
this.deathTimer = 60;   // 60 秒复活倒计时
this.reviveTimer = 60;
```
但 `syncSprite()` 用**固定常量 0.6** 计算淡出进度：
```js
const progress = Math.max(0, this.deathTimer / 0.6);  // 60 / 0.6 = 100
this.sprite.setAlpha(progress);  // clamp 到 1，永远满透明度
```

普通敌人 `deathTimer=0.6`，progress 从 1→0 正常淡出；Boss 的 `deathTimer=60`，progress 恒为 100（clamp 1），**60 秒内 Boss 尸体完全不透明、不缩放、不旋转**，直到复活。这与「倒下积蓄力量」的视觉预期严重不符。

**修复建议**：Boss 死亡用独立的淡出时长，或分离「淡出动画时长」与「复活等待时长」：
```js
// takeDamage 中 Boss 首次死亡
this.deathTimer = 0.6;        // 淡出动画时长
this.reviveTimer = 60;        // 复活等待
// syncSprite 中
const fadeDur = this.type === 'boss' ? 0.6 : 0.6;
const progress = Math.max(0, this.deathTimer / fadeDur);
// 死亡动画播完后保持隐身等待复活
if (this.reviveTimer > 0 && this.deathTimer <= 0) {
  this.sprite.setAlpha(0);
  return;
}
```

### 1.3 `package.json` phaser 版本与历史报告、运行配置不一致

**位置**：`client/package.json:19`（`"phaser": "^3.80.1"`）、`GameApp.js:12`（`Phaser.CANVAS`）

**问题**：
- 历史审计报告（2026-06-18）多次声称「Phaser 3（v3.90.0）」，但 `package.json` 声明 `^3.80.1`，实际安装版本未锁定（`node_modules` 当前未安装，`package-lock.json` 存在但未核验）。两份报告的版本陈述与依赖声明矛盾。
- `GameApp.js` 使用 `Phaser.CANVAS`（Canvas 渲染），而非 `Phaser.AUTO`/`Phaser.WEBGL`。Canvas 模式下不支持 WebGL 着色器、混合模式效果有限，粒子/光效表现会打折。历史报告「视觉品质」相关结论基于此渲染模式，需明确告知。

**修复建议**：
1. 统一版本陈述，将 phaser 锁定到确定版本（去掉 `^` 或更新到目标版本）。
2. 评估改用 `Phaser.AUTO`（优先 WebGL，回退 Canvas）以提升特效表现。

## 2. 较高严重度问题 🟠

### 2.1 暂停栈在「Esc 暂停」与「对话/升级」并存时失效

**位置**：`GameController.js:127-165`、`GameController.js:385-402`

**问题**：`update()` 中暂停逻辑：
```js
if (!this.paused) {
  // ... 处理 Esc、Tab、玩家更新、敌人更新
}
```
当对话（`addPause('dialogue')`）或升级（`addPause('levelup')`）触发后 `this.paused=true`，此时 `if(!this.paused)` 块整体跳过，**Esc 键无法响应**。玩家在对话/升级面板打开时按 Esc 无任何反馈。

更隐蔽的是：`togglePause()` 内部 `if (this.levelUpOpen || !this.running) return;` 会阻止升级时暂停，但**不阻止对话时暂停**——然而由于上述 `if(!this.paused)` 包裹，对话期间 Esc 根本进不到 `togglePause`。两条路径行为不一致。

**修复建议**：将暂停/装备切换键的检测移到 `if(!this.paused)` 之外，或单独处理「用户请求退出当前模态」的逻辑。

### 2.2 `DropManager` 拾取双通道，`KeyE` 拾取与自动拾取语义重叠

**位置**：`GameController.js:131-133`（KeyE 拾取）、`DropManager.js:45-56`（autoPickupDrops）、`DropManager.js:58-91`（pickupDrop）

**问题**：
- `autoPickupDrops` 在 `update` 中每帧扫描，pickupRadius = `player.radius + 24` = 60，自动拾取近距离掉落。
- `KeyE` 调用 `pickupDrop()`（无参）拾取 `nearestDrop`，nearestDrop 判定半径 120。
- 当 nearestDrop 同时在 60 半径内时，`autoPickupDrops` 已将其从 `this.drops` 移除；随后 `pickupDrop(null)` 走 `if(!d) return`，安全但 `KeyE` 完全失效——玩家按 E 无反馈。
- 当 nearestDrop 在 60~120 之间时，`KeyE` 才真正生效。但此时玩家可能误以为「靠近就能捡」，UI 的 `pickupHint`（基于 nearestDrop）显示「按 E 拾取」，实际自动拾取不会触发，造成认知混乱。

**修复建议**：明确单一拾取机制。要么全程自动拾取（删除 KeyE），要么 KeyE 拾取且取消自动拾取，或调整两者半径使语义清晰。

### 2.3 Boss 复活数值翻倍可能溢出，且 `onBossRevived` 重复满血

**位置**：`Enemy.js:187-212`（reviveBoss）、`GameController.js:265-272`（onBossRevived）

**问题**：
```js
// reviveBoss
this.maxHp = Math.floor(this.baseMaxHp * 2);
this.hp = this.maxHp;
this.atk = Math.floor(this.baseAtk * 2);
// ...
// onBossRevived
boss.hp = boss.maxHp;  // 重复赋值，无害但冗余
```
1. `reviveBoss` 后 `baseMaxHp/baseAtk/baseDef` 被重新赋值为翻倍后的值。若 Boss 理论上可二次复活（当前 `hasRevived=true` 阻止，但逻辑耦合脆弱），数值会指数增长。
2. `onBossRevived` 的 `boss.hp = boss.maxHp` 是死代码，`reviveBoss` 已设满血。
3. enhanced Boss（`sizeScale=1.4`）复活后 `radius = baseRadius * sizeScale` 用的是 1.4，但 enraged 后 `sizeScale=1.5`，复活未重置 enraged 状态，可能产生尺寸不一致。

**修复建议**：清理冗余赋值；明确复活后的 enraged/sizeScale 状态；将「翻倍」改为基于原始 `baseMaxHp` 的固定倍数，避免链式放大。

### 2.4 `Player.takeDamage` 缺少 `game` 参数时静默无特效

**位置**：`Player.js:436-455`、`Enemy.js:306`（`p.takeDamage(Math.floor(this.atk * 1.5))`）、`Enemy.js:315`（`p.takeDamage(this.atk)`）

**问题**：`Enemy.update` 中两处玩家受击调用 `p.takeDamage(dmg)` **未传 `game`**：
```js
p.takeDamage(Math.floor(this.atk * 1.5));  // 缺 game
p.takeDamage(this.isSkill ? ... : this.atk);  // 缺 game
```
`takeDamage` 签名 `takeDamage(dmg, game = null)`，`game=null` 时跳过所有受击特效（伤害数字、粒子、震屏、闪光）。玩家被敌人近战/大招击中时**没有任何受击视觉反馈**，仅有 hurt 动画和闪烁。

对比 `CollisionManager.js:28` 的投射物命中正确传了 `game`。

**修复建议**：`Enemy.update` 两处改为 `p.takeDamage(dmg, game)`。

## 3. 一般严重度问题 🟡

### 3.1 `Enemy.update` 击退位移计算错误，速度被 dt 双重计入

**位置**：`Enemy.js:238-243`

```js
if (this.hitKnockbackTimer > 0) {
  this.hitKnockbackTimer -= dt;
  const kb = this.hitKnockbackTimer > 0 ? 120 * this.hitKnockbackTimer : 0;
  this.x += Math.cos(this.hitKnockbackDir) * kb * dt;
  this.y += Math.sin(this.hitKnockbackDir) * kb * dt;
}
```

**问题**：`kb` 已是「剩余时间 × 120」的衰减量，再乘 `dt` 导致击退速度随 dt 平方衰减，且总位移远小于设计值（0.12s 内约 120×0.12×0.12/2 ≈ 0.86 像素，几乎不可见）。正确应是 `kb` 作为速度直接乘 dt，或用积分形式。

**修复建议**：
```js
const kbSpeed = 120 * (this.hitKnockbackTimer / 0.12); // 0~120 速度
this.x += Math.cos(this.hitKnockbackDir) * kbSpeed * dt;
```

### 3.2 `Player.useSkill` 命中判定使用 `this.dir`，鼠标瞄准时方向突变

**位置**：`Player.js:390-397`

```js
for (const e of game.enemies) {
  // ...
  if (Math.abs(angleDiff(angleTo, this.dir)) <= sk.arc / 2) hits.push(e);
}
```

**问题**：技能命中扇形基于 `this.dir`。当玩家移动时 `this.dir` 由移动方向决定；当鼠标按住瞄准时 `this.dir` 由鼠标方向决定。但 `useSkill` 在 `update` 末尾响应按键，此时 `this.dir` 是**当前帧更新后的方向**。若玩家正在移动并同时按 J，技能朝移动方向释放而非角色面朝方向，与「鼠标瞄准」的预期冲突（移动中鼠标瞄准会被移动方向覆盖，见 `Player.js:258` `if(!this.mouseAim) this.dir = ...`）。

**修复建议**：明确「技能方向」与「移动方向」分离，或鼠标按住时锁定技能方向为鼠标方向。

### 3.3 `Particle` 对象池未真正复用 sprite，每次 reset 销毁重建

**位置**：`Particle.js:30-46`、`EffectManager.js:14-18`

**问题**：`ObjectPool` 复用的是 `Particle` 对象，但 `Particle.reset` 每次都 `this.sprite.destroy()` 再新建 sprite：
```js
if (this.sprite) { try { this.sprite.destroy(); } catch(e){} this.sprite = null; }
// ... 新建 image 或 graphics
```
这使得对象池**只省了 Particle JS 对象的分配，sprite 仍每帧创建销毁**，GC 压力未真正缓解。历史报告声称「粒子高频创建销毁已用对象池优化」，但实际优化效果有限。

**修复建议**：池化的 Particle 持有持久 sprite，reset 时只 `setTexture`/`setVisible(true)`/重置位置，避免销毁重建。或改用 Phaser 原生 `ParticleEmitter`。
